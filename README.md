#Конфигурационное управление
#Этап 1
feat(repl): реализован базовый GUI-интерфейс эмулятора (Этап 1, Вариант №12)

- Приложение реализовано в виде графического интерфейса (Tkinter)
- Заголовок окна формируется как "Эмулятор-[username@hostname]"
- Парсер корректно обрабатывает аргументы в кавычках (с использованием shlex.split)
- Реализованы команды-заглушки:
  • ls → выводит "ls [аргументы]"
  • cd → выводит "cd [аргументы]"
  • exit → завершает приложение
- Все ошибки (неизвестная команда, синтаксис) перехватываются и отображаются
- Продемонстрирована работа в интерактивном режиме с примерами:
  > ls /home
  > cd "my folder"
  > unknown → "Ошибка: неизвестная команда 'unknown'"

Соответствует требованиям Этапа 1 Варианта №12.

#Этап 2
feat(config): реализована настройка через CLI и YAML (Этап 2, Вариант №12)

- Поддержка параметров командной строки:
  • --vfs-path
  • --log-path
  • --script-path
  • --config-path
- Конфигурационный файл в формате YAML с теми же параметрами
- Приоритет: CLI > YAML (значения из командной строки переопределяют файл)
- Логирование вызовов команд в CSV-файл с полями:
  timestamp, command, success, error_message, username
- Стартовый скрипт:
  • Поддерживает комментарии (#)
  • Останавливается при первой ошибке
  • Отображает "[Скрипт:N] > команда" при выполнении
- Обработка ошибок:
  • Невозможно прочитать конфиг → предупреждение
  • Ошибка в скрипте → остановка + сообщение

Протестировано на нескольких сценариях запуска.
Соответствует требованиям Этапа 2 Варианта №12.

#Этап 3
feat(vfs): интеграция виртуальной ФС из директории (Этап 3, Вариант №12)

- VFS загружается рекурсивно из физической директории (без архивов)
- Все операции выполняются только в памяти (никакой записи на диск)
- При отсутствии --vfs-path создаётся VFS по умолчанию:
  /home, /etc (с motd), /bin, /tmp
- Поддержка motd: при старте выводится содержимое /etc/motd (если существует)
- Команда vfs-info выводит:
  • Имя VFS (имя исходной директории или "default_vfs")
  • SHA-256 хеш структуры VFS (в JSON-представлении)
- Обработка ошибок: отсутствующая директория → исключение

Протестировано на:
- Пустой VFS
- VFS с 3+ уровнями вложенности
- Отсутствующем пути

Соответствует требованиям Этапа 3 Варианта №12.

#Этап 4
feat(commands): реализованы ls, cd, whoami, who (Этап 4, Вариант №12)

- Команда ls:
  • Отображает содержимое текущей или указанной директории
  • Работает с относительными и абсолютными путями
- Команда cd:
  • Поддерживает переходы по .., /, относительным путям
  • Корректно обновляет текущую директорию
- Команда whoami:
  • Возвращает имя текущего пользователя ОС (os.getlogin())
- Команда who:
  • Выводит список пользователей: admin, текущий, guest + hostname

Все команды:
- Интегрированы в GUI
- Поддерживают выполнение из стартового скрипта
- Логируются в CSV

Добавлен тестовый скрипт с примерами всех команд.
Соответствует требованиям Этапа 4 Варианта №12.

#Этап 5
feat(vfs-mutation): реализованы chown и rm (Этап 5, Вариант №12)

- Команда chown <владелец> <путь>:
  • Устанавливает поле "owner" у файла/директории в VFS
  • Работает с существующими путями
  • Возвращает понятное сообщение об успехе/ошибке
- Команда rm <путь>:
  • Удаляет файл или директорию из VFS (только в памяти)
  • Не затрагивает физическую ФС
  • Корректно обрабатывает несуществующие пути

Обе команды:
- Полностью соответствуют требованиям Этапа 5 Варианта №12
- Интегрированы в GUI и стартовый скрипт
- Логируются в CSV (включая ошибки и имя пользователя)

Примеры в стартовом скрипте:
  chown admin /etc/motd
  rm /etc/motd

Тестирование:
- Успешное изменение владельца
- Удаление существующего файла → успех
